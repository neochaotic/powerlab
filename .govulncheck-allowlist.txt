# scripts/govulncheck-gate.sh blocks CI on any reachable, non-stdlib vuln that
# is NOT listed here. This file is the ratchet: the gate catches NEW regressions
# (the next viper-class boot break) while the known backlog is tracked below.
# Shrink this list as remediations land; never grow it without a tracking issue.
#
# Format: one OSV id per line (first whitespace-delimited token). Everything
# after is a comment: module — reason — tracking.
#
# Stdlib CVEs are NOT listed here: the gate ignores them by design (they are
# handled by keeping the Go toolchain current via CI's floating go-version).

# ── moby/docker stack — residual after the docker v28 / compose-go v2
#    migration (#523). grpc (v1.79.3), containerd/v2 (v2.2.1) and otel/sdk
#    (v1.40.0) were independently bumpable without disturbing the docker
#    v28.5.1 pin, so GO-2026-4762 / GO-2025-4108 / GO-2025-4100 / GO-2026-4394
#    are now CLEARED. All FIVE entries below converge on ONE upstream event:
#    the docker v28→v29 / `moby/moby` v2 rename. buildkit's fix (v0.28.1)
#    pulls docker/cli v29; compose's fix is cli v29.2; docker/docker's fix is
#    moby/moby v2. compose/v2 v2.40.3 (latest) is still built against docker
#    v28.5.1, so forcing any of them creates a compose↔docker v28/v29 API
#    skew. When compose ships a docker-v29 build, all five clear together —
#    .github/workflows/compose-v29-watch.yml polls for that and pings #523.
#
# Exposure note for the docker/docker pair: both are DAEMON-side bugs in the
# dockerd AuthZ-plugin path (4887: AuthZ bypass on oversized bodies; 4883:
# off-by-one in plugin-privilege validation). The OSVs carry no symbol info,
# so govulncheck flags the whole package as "called" — but app-management is
# an API *client*; it does not run an authorization plugin in-process. Real
# exposure requires running dockerd with `--authorization-plugin`, which
# PowerLab neither ships nor configures. Tracked, low-risk, fixed in moby v2.
