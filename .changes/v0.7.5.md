## [v0.7.5] — 2026-05-31
### Added
- powerlab-mcp MVP — talk-to-your-homelab surface for AI agents. Read-only resources (system://metrics, journal://, audit://) over the official MCP Streamable HTTP transport, gated by your PowerLab login. Three MVP guardrails ship with it: (1) operator kill-switch `Disabled = true` in /etc/powerlab/mcp.conf — service exits 0 before binding, no systemctl mask needed; (2) dead PIDFile= + Environment=HOME= removed from powerlab-mcp.service (binary doesn't write a pidfile and doesn't need HOME); (3) end-to-end smoke client in backend/powerlab-mcp/cmd/smoke that lists + reads every advertised resource via the official Go SDK — usable as a pre-flight before any release cut. Plus: deep architecture documentation at docs/concepts/mcp-server.md (Mermaid topology + auth flow + journey-of-a-read, test recipes for loopback/LAN/Go, Claude Desktop & Cursor & Code wire-up, defense-in-depth matrix, roadmap) and a README "Talk to your homelab" section that positions MCP as a complementary surface, NOT a product pivot. Tests: config TestLoad_DisabledKey (truthy spellings + safety defaults), main_test TestRun_DisabledKillSwitchExitsCleanly (proves run never binds when Disabled), check-package-linux-powerlab-mcp_test.sh extended to lock the Disabled key into mcp.conf.sample and lock PIDFile / Environment=HOME out of the unit.
- powerlab-mcp Phase 2 foundation per ADR-0044 — hybrid architecture pivot. Audit + journal of PowerLab units stay independent (survive core being down); sysadmin telemetry (planned: system://utilization, disk, network, GPU, apps://, docker://) will be thin proxies to core/app-management endpoints, eliminating ~80% of the duplication the original "isolated MCP" plan was about to ship. This PR lands the foundation: ADR-0044 documenting the pivot; coreproxy package with URL discovery (reads core's .url file under RuntimePath), 10s URL cache, JWT forwarding, structured core_unavailable errors with fallback hints; docs://api manifest + docs://api/{service} raw OpenAPI YAML for every PowerLab service (Scalar-equivalent surface for agents — agent can self-discover the entire PowerLab API without us hand-coding each endpoint as an MCP tool); journal://units discovery resource so an agent reads available powerlab-*.service stems instead of guessing. Config gains OpenAPIDir + SystemdSystemDir keys; constructor refactored to a resourcesConfig bag so future proxy resources don't balloon the signature. Path-traversal hardened on docs://api/{service}. 19 new tests across coreproxy, journal/units, server/resources_docs.
- powerlab-mcp system telemetry trifecta — system://disk, system://network, system://gpu, plus a system://schema covering the whole namespace. ADR-0044 hybrid pattern: system://disk + system://network thin-proxy core (/v1/sys/disk + /v1/sys/net) so the agent reads the same data the panel dashboard does — including SMART metadata and per-interface state — without us re-implementing /proc parsers. system://gpu imports common/external::GetGPUUtilization directly (same code path as the panel's GPU widget): Apple Silicon via ioreg, Nvidia via nvidia-smi, "no GPU detected" returns an empty model string rather than an error. system://schema is the agent's discovery index — describes every resource (independent vs proxied), the field shapes per resource, and the canonical core_unavailable proxy-error shape. Smoke client extended to assert each new resource (proxied: WARN on core_unavailable, fallback hint check; GPU: shape contract + percent in 0..100); assertSchemaPayload relaxed to accept both the single-fields pattern (audit/journal) and the per-resource fields_* pattern (system). resources/list now advertises 10 resources. Live-verified on macOS: GPU detected as Apple M5 (46% util, 538 MiB used).
- powerlab-mcp apps://* + docker://logs/{id} implementation per ADR-0045 — storage-agnostic thin proxies to app-management's HTTP /v2/ API. coreproxy.Client generalised to multi-service registration with per-service URL caching (core via casaos.url, apps via app-management.url); error codes follow <service>_unavailable / <service>_status_NNN so the agent pattern-matches uniformly across upstreams. Resources shipped: apps://schema (discovery), apps://list, apps://state/{id} + four sub-templates (containers/health/stats/disk), docker://logs/{id} (proxied through ComposeAppLogs so MCP NEVER accesses the Docker socket — ADR-0045 win #2). powerlab-mcp.service gains Wants=powerlab-app-management.service (soft dep — MCP still boots when app-management is down; resources serve the canonical apps_unavailable payload). resources/list now advertises 12 resources. Smoke client extended: dispatch covers apps://* + docker://* via the same assertProxiedPayload (generalised isDegradedCode helper recognises any <service>_unavailable / _status_ shape uniformly); assertSchemaPayload relaxed to accept a "resources" routing map (proxy schemas — field shapes live in docs://api/<service>) alongside fields / fields_*. Legacy Client.Get and Client.ResolveCore wrappers removed; all callers migrated to GetFrom(ctx, ServiceCore, ...) and Resolve(ServiceCore). Deadcode clean. The architectural promise locked: when PowerLab migrates from SQLite to PostgreSQL, MCP needs zero changes.
- powerlab-mcp first batch of curated tools per ADR-0046 — read-only tier dogfoods the tool-call pattern without risk. journal_search wraps the existing journal:// reader with a literal substring filter + line cap (passes the unit verbatim through canonicalUnit so the agent cannot escape PowerLab scope); check_disk_free wraps syscall.Statfs at any path defaulting to / (returns total/available/used bytes + percent in a typed shape). Both ship with hand-written JSON Schemas (auto-derived from struct tags) and descriptions tuned for LLM ergonomics with the side-effect class explicit ("READ ONLY") so the model surfaces it to the user. Five new tests covering: tools/list advertises both, descriptions carry the side-effect class, journal_search filters correctly + requires unit, check_disk_free returns the shape contract + rejects missing paths with a friendly message. Live-verified against the Mac dev binary: tools/list returns 2 tools; check_disk_free / reports 70.8% used on the host. Deadcode clean. resources/list unchanged (still 12) — tools are a parallel namespace. Future destructive tools (ADR-0046 batch 3) will register from a separate gated path when EnableDestructiveTools lands.
- powerlab-mcp tools batch 2 — restart_app (ADR-0046 reversible side-effect tier). Cycles every container of one installed PowerLab app via a PUT to app-management's /v2/app_management/compose/{id}/status with body '"restart"' (the spec's enum encoding). Not gated on EnableDestructiveTools because the blast radius is bounded — containers go down briefly then come back up, no data loss, app ends in the same state. Description leads with "SIDE EFFECT" so Claude Desktop / Code surface the marker in the tool-use prompt. coreproxy gained RequestFrom(method, service, path, token, body, contentType) — verb-and-body capable for the write-class tools landing in ADR-0046 batches 2 + 3; GetFrom now thin-wraps it (backwards compat for every existing call site). Architecture note: MCP calls app-management direct via coreproxy bypassing the gateway, so the gateway-written audit.jsonl does NOT capture MCP-driven restarts. Full audit correlation lands with the MCP audit-recorder dogfood (ADR-0034 deferred item) — until then the agent's action trail is in journal://mcp. 8 new tests: RequestFrom round-trips verb+body+content-type+UA; non-2xx surfaces with verb in Detail; GetFrom backwards compat; tool advertised; tool description carries SIDE EFFECT class; happy-path PUTs the exact wire shape app-management expects (verb + path + Content-Type + body verbatim); 404 surfaces as apps_status_404 + IsError; app-management down surfaces apps_unavailable + IsError; invalid ids (empty, path-traversal, dotted, query-shaped) rejected at the tool layer before reaching the proxy.
- powerlab-mcp composevalidator package — deny-list compose YAML validator per ADR-0046 §4, the prerequisite for the install_app tool (batch 3). Inspects services for: privileged: true (container escape), host namespace sharing (network_mode/pid/ipc/uts/userns_mode: host plus network_mode: container:<id>), dangerous cap_add (SYS_ADMIN, NET_ADMIN, ALL, etc. — case-insensitive, tolerates CAP_ prefix), raw /dev/* device passthrough, and bind mounts to sensitive host paths (/var/run/docker.sock + variants, /proc, /sys, /etc, /root, /var/lib, /var/log, /dev, /boot, library + binary dirs). Returns a typed Result with per-violation Code + Detail + Service so an agent and the operator-facing report read the same shape. Plus: powerlab-mcp-validate CLI (cmd/validate) reads a YAML from path or stdin and reports human-readable text or JSON, exits 0/1/2 — operators can run it standalone before any MCP involvement. 24 tests cover a realistic PowerLab-style happy path, every documented violation (19 negative table-driven cases), invalid YAML, service-less doc, compound violations on one service, per-service offender attribution. Live-verified: malicious compose with privileged: true + network_mode: host correctly flags both. Validator is deny-list-only, not a full compose-spec parser (app-management still validates syntax + image policy); ADR-0046 §4 names it as one layer among many — install_app additionally gates on EnableDestructiveTools opt-in or panel approval flow.
- powerlab-mcp tools batch 3 — install_app + uninstall_app (ADR-0046 destructive tier), gated on the new EnableDestructiveTools mcp.conf knob (default false). install_app runs composevalidator.Validate against the proposed compose YAML BEFORE forwarding to app-management — if any ADR-0046 §4 deny-list pattern trips, the upstream is never called and the agent reads a structured violations payload. Supports dry_run=true (forwards the same flag to app-management for upstream-side syntax + image policy validation). uninstall_app DELETEs /v2/app_management/compose/{id}. Both tools' descriptions lead with "DESTRUCTIVE" so Claude clients surface the marker in the tool-use prompt. registerDestructiveTools is only called when rc.enableDestructiveTools is true — when false the tools are NOT registered at all (don't appear in tools/list; agent has no way to call them). The opt-in is documented as the operator's explicit threat-model consent in mcp.conf.sample. ADR-0046 originally named prune_orphans in this batch; it's deferred to #619 because no matching app-management HTTP endpoint exists today (only the boot-time MigrateOrphanCuratedApps internal migration). 10 new tests: gate (NOT advertised when flag false / advertised + DESTRUCTIVE marker when true), install_app (validator-rejects-before-reaching-upstream / valid YAML POSTs with Content-Type application/yaml verbatim body / dry_run query param forwarded / required compose_yaml validation), uninstall_app (DELETE wire shape / invalid id rejection at tool layer). config Load tests pin truthy spellings + default-false safety. Regression test in scripts/check-package-linux-powerlab-mcp_test.sh locks the EnableDestructiveTools = false sample.
- cmd/smoke now exercises the MCP tool surface alongside the resource surface — probeAndCallTools reads tools/list and asserts every advertised tool's Description carries an explicit side-effect class marker (READ ONLY / SIDE EFFECT / DESTRUCTIVE) per ADR-0046 §1; verifies the unconditional tools (journal_search, check_disk_free, restart_app) are advertised; tolerates install_app + uninstall_app being either present-and-DESTRUCTIVE-marked (when EnableDestructiveTools=true) or absent (the default-off gate). The two read-only tools are driven end-to-end with real arguments and their typed outputs validated: journal_search against unit=gateway returns an entries array with the right field shape (degraded box surfaces a WARN, not a hard fail — the tool wiring is what matters); check_disk_free against / asserts used+available==total, 0<=used_percent<=100, total>0. probeFirstOpenAPISpec reads docs://api dynamically, picks the first staged service, and validates its OpenAPI spec shape (starts with openapi:/swagger:, > 200 bytes); an empty manifest surfaces as WARN (Mac dev / pre-install — wire-up fine, staging not run) rather than a failure. decodeStructured helper round-trips StructuredContent through JSON so every typed-output check stays one call site. Live-verified on macOS: 3 tools advertised + gate respected for install_app/uninstall_app; check_disk_free / reports 71.5% used / 134 GiB available; system://gpu returns Apple M5 38% util.
### Fixed
- **MCP — `system://network` now hits the real core path.** Resource was
proxying to `/v1/sys/net` (a stale godoc annotation), which doesn't exist
on any deployed core and returned `core_status_404`. Corrected to
`/v1/sys/network/interfaces`, the actual route registered in
`core/route/v1.go`. Caught by the live battery on .142, where the smoke
client now reads real interface enumeration (name/IP/MAC/type/state) for
every NIC instead of an error payload.

- Settings → About: when you click "Check now" and the update check fails, the card now shows the reason — "HTTP 401 — your session expired", "the host is unreachable", or the gateway's status code — instead of falling through to the silent "Background check did not complete" text, which read as "no update available". A user-initiated check always surfaces its outcome; a single failed background poll still stays quiet.
- Settings → About: clicking "Check now" again right after a failed check no longer flashes "Update check failed — ." (an empty reason) while the new request is in flight. The manual-failure line now clears the moment a re-check starts and only reappears if that re-check also fails.
- ci: install-smoke gate was chronically red post-merge (13/20 recent main pushes failed) because the CI tarball is stamped 0.0.0-ci and install.sh refuses 0.0.0-* dev builds by default. The workflow now sets POWERLAB_ALLOW_DEV_BUILD=1 so the gate actually exercises a real install. PRs skip install-smoke entirely (the job is guarded `if: github.event_name != 'pull_request'`), so the silent post-merge red was invisible — no recent unit-file change has actually been validated end-to-end. Regression test scripts/check-ci-install-smoke-dev-build_test.sh locks the env var into the workflow.
- powerlab-mcp: audittail no longer OOMs the service on a large audit log. The original implementation read the WHOLE audit.jsonl into a single []byte before scanning — a 500 MiB audit log (a real-life scenario on a long-running chatty box) drove ~2.8 GiB of RSS during a single audit:// read, which would OOM a Pi 4 / 4-8 GiB box. Recent + ByCorrelation now stream the file via bufio.Scanner with a ring buffer of the requested limit, so peak memory scales with the limit (max 1000 records, a few hundred KiB) regardless of file size. Adversarial regression test TestRecent_MemoryBoundOnLargeFile synthesises a 200 MiB audit log and asserts heap delta stays under 50 MiB (was ~600 MiB+ with the old code). Plus: audit://schema completeness — ts_us / query / kind fields were absent from the schema doc despite being in every record (operators + agents reading the schema would have missed three fields); now documented with the same rigor as the rest. Plus: cmd/smoke gains per-resource data-quality assertions (audit ts must be RFC 3339, status in HTTP range, method a known verb, remote_ip non-empty; system metrics shape locked field-by-field). Smoke now fails when a future regression breaks payload shape, not just when a read errors. Minor findings filed: #606 (body-limit returns 400 not 413) and #607 (/version info disclosure).
- powerlab-mcp packaging integrity (ADR-0044 follow-up to #609). The hybrid-architecture foundation merged in #609 had five integrity gaps that would have left features shipped-but-broken in production: (1) package-linux.sh did not stage the OpenAPI specs so docs://api returned an empty manifest on a real install; (2) install.sh did not create /usr/share/powerlab/openapi/ or copy specs into it; (3) mcp.conf.sample did not document the new OpenAPIDir + SystemdSystemDir keys so operators could not discover them; (4) powerlab-mcp.service did not soft-depend on powerlab-core.service even though ADR-0044 promised it; (5) cmd/smoke did not probe system://utilization at all, so a future regression breaking the proxy pattern would have shipped silently. This PR closes all five. Also drops a left-over dead _docsTestImport function. Regression test check-package-linux-powerlab-mcp_test.sh extended with 6 new assertions covering each gap. Smoke client gains assertProxiedPayload — treats core_unavailable as WARN with operator guidance ("audit + journal still readable"), real upstream payload as PASS, missing fallback hint as FAIL.
### Security
- Bump github.com/moby/spdystream v0.5.0 → v0.5.1 in app-management to clear GO-2026-4958 (CVE-2026-35469): an unauthenticated remote-DoS in the SPDY frame parser that allocates attacker-controlled amounts of memory from a single crafted frame. Reachable transitively via docker/compose → buildx's k8s driver; the patch adds bounds checks on SETTINGS counts and header lengths.
### Internal
- Revise and accept ADR-0034 (standalone observability + MCP service). Reconciled against current code: binary name fixed to `powerlab-mcp`; `audit://` resources now tail the JSONL audit store (ADR-0035, shipped) instead of attaching SQLite; transports phased to HTTP+SSE first (stdio deferred); roadmap re-keyed from "Sprint 17" to release increments; JWT confirmed single-issuer via the user-service. Host-ops scope tightened to PowerLab-scoped tools only (no arbitrary kill_process / list_directory) under the enterprise lens.
- Scaffold the `powerlab-mcp` service module (ADR-0034 Foundation): a CGO-free binary serving the control endpoints (/healthz, /version) and the MCP Streamable-HTTP transport at /mcp over HTTP+SSE on :9090, with a forgiving `mcp.conf` loader (missing file → defaults so a fresh box boots). No resources or tools registered yet; CI builds, tests, deadcode- and vuln-scans the module. Not yet wired into the installer — it ships to users with the first real resources.
- powerlab-mcp: gate the MCP endpoint at the read tier (ADR-0034). Loopback (the trusted local agent) is free; a LAN caller must present a valid PowerLab user-service JWT, validated via the shared `jwt.HTTPJWT` gate against the JWKS published under the configurable RuntimePath. The /healthz and /version control endpoints stay open. Loopback never attempts JWT key resolution, so local observability keeps working when the user-service is unreachable.
- powerlab-mcp (self-review hardening): a request that claims to come from loopback but carries reverse-proxy headers (X-Forwarded-For / Forwarded / X-Real-Ip) no longer inherits the loopback auth skip — it is forced through the JWT check. Closes a latent auth-bypass had the standalone service ever been fronted by a same-host reverse proxy. Genuine local agents (no proxy headers) keep zero-config trust; authenticated proxied callers still pass.
- powerlab-mcp: first MCP resource — system://metrics serves a point-in-time host snapshot (memory total/available/used%, load average 1/5/15m, uptime) read directly from /proc, independent of the rest of PowerLab. Adds an in-process MCP integration test (initialize → resources/list → resources/read) proving the protocol works end-to-end, not just that the endpoint is reachable.
- powerlab-mcp (milestone review follow-up): system://metrics now reports CPU core count, so an agent can interpret the load averages (load relative to cores = utilisation). Adds an MCP integration test over the real HTTP Streamable transport (not just the in-process client), proving the transport and protocol compose end-to-end.
- powerlab-mcp (A+ code review fixes): a bind failure (e.g. port already in use) now exits non-zero instead of a silent clean stop systemd would misread — the lifecycle was extracted into a testable run() that binds synchronously and only signals systemd READY after the bind succeeds. Loopback detection now uses net.SplitHostPort so IPv6 "[::1]:port" is recognised. Minor: round2 via math.Round.
- powerlab-mcp (meta-review hardening): cap the /mcp request body at 1 MiB via http.MaxBytesReader. The MCP transport reads the whole body into memory, so an unbounded POST was an OOM/DoS vector — sharper because loopback callers are unauthenticated. Legitimate JSON-RPC requests are a few KB, so the cap is invisible to real use.
- powerlab-mcp: add the journal:// resource — systemd journal entries (time/unit/priority/message) via journalctl, plus a self-describing journal://schema. Access is scoped to PowerLab units (powerlab-*.service): the agent-supplied unit is canonicalised, not passed through, and args go to exec as a literal argv (no shell), so there is no flag/shell-injection or cross-service log-read surface. Query params lines/since/priority via an RFC 6570 template.
- powerlab-mcp journal:// review fixes: cap the line count (default 200, ceiling 2000) so a query with no `lines` no longer dumps the entire journal since boot (token/memory blowout); and default a record with no PRIORITY field to info (6) instead of emergency (0), which had inflated every unprioritised line to the top severity.
- powerlab-mcp: migrate from the third-party mark3labs/mcp-go (v0.54.x, pre-1.0) to the official modelcontextprotocol/go-sdk (v1.x, GA, maintained with Google/Anthropic, tracks the current spec). Done while the surface was still small (system:// + journal://, no tools): server construction, both resources, the streamable-HTTP mount, and the MCP integration tests re-ported; the pure-logic packages (metrics/journal/config) are untouched. ADR-0034 amended (open question Q6 resolved).
- powerlab-mcp: add the audit:// resource — read-only tail of the gateway's JSONL audit log (ADR-0033/0035), exposing audit://schema (self-describing), audit://recent{?limit} (newest records, default 100 / max 1000), and audit://action/{correlation_id} (records tagged with one X-Request-Id, e.g. everything one request triggered). Reads a single file (the gateway is the only audit writer) — ADR-0034 amended to match reality. With this, the read-only observability surface is complete: an MCP agent reads metrics + logs + audit of the box.
- powerlab-mcp: wire into release packaging (#599). scripts/package-linux.sh now cross-compiles powerlab-mcp (CGO-free, amd64+arm64) with a -X main.version=$VERSION stamp, emits /etc/powerlab/mcp.conf.sample (ListenAddr/AuditDir/RuntimePath) and powerlab-mcp.service (Type=notify, User=root because audit.jsonl is root:root 0600, ExecStart uses -conf to match the binary's flag), and adds mcp to install.sh + uninstall.sh SERVICES lists so systemctl enable/restart/stop iteration covers it. ADR-0034 packaging acceptance items checked off. Regression test scripts/check-package-linux-powerlab-mcp_test.sh locks every contract the change had to honour; ldflags regression test gets a new positive assertion for main.version.
- powerlab-mcp: amend ADR-0034 to drop the three-tier auth fiction. The stack today has no Role claim on the JWT and every user is hardcoded admin at registration (user-service/route/v1/user.go:65,140), so a third "admin" tier would be doc-theatre without enforcement. The actual model is two-tier: loopback (trusted local) / authenticated (any valid PowerLab JWT). Tools added later inherit the same gate — any authenticated caller can call any tool until real RBAC lands (backlog #603). Updates ADR-0034 table + rationale + acceptance checklist; updates backend/powerlab-mcp/server/server.go package doc to match.

No new Docker socket dependency in powerlab-mcp.service; just a Wants= soft dep on powerlab-app-management.service. ADR published in English alongside ADR-0034/0044.
