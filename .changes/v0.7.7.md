## [v0.7.7] — 2026-06-02
### Added
- powerlab-mcp now serves OAuth 2.1 discovery (RFC 9728 / RFC 8414 / RFC 7591 /.well-known endpoints) plus a minimal authorization-code + PKCE-S256 flow so standard MCP clients (Claude Desktop via mcp-remote, Claude Code's HTTP transport, the Anthropic MCP SDK) can complete the 2025-03-26-spec handshake instead of aborting on the prior 404. The underlying auth model is unchanged — loopback callers still bypass per ADR-0034, LAN callers still need a real user-service JWT — but the discovery façade lets agents written to spec reach `/mcp` at all.
- Three new MCP Tools wrap PowerLab's authoring guidance so chat-mode agents that only surface Tools can reach it: `browse_catalog` (list catalog apps, with optional substring filter), `get_compose_conventions` (the canonical conventions markdown), `start_compose_authoring` (the curated bundle of conventions + 3 catalog examples + composevalidator deny-list — Tool form of the existing compose_authoring Prompt primitive). Pre-fix the only paths to this content were Prompts (which most chat clients don't autonomously discover) and templated Resources (which require URI navigation an agent may not know to attempt); post-fix tools/list advertises them directly. (P0.1 from the 2026-05-31 MCP-only chat-mode test.)
- New `get_system_health` MCP Tool aggregates memory + disk + PowerLab services + pending updates into one structured response with per-category severities (ok | warn | critical | unknown) and an overall verdict that escalates to the worst component. Pre-fix, an agent asked "how's the system health" had to read four separate system:// resources and encode the threshold correlations itself — the smoke client already had this knowledge baked in but agents in chat mode didn't. Thresholds match the panel's health dashboard so an agent's surfaced severity tracks what an operator sees in the UI. (P1.5 from 2026-05-31 MCP-only chat-mode test.)
- New `generate_artifact` MCP Tool gives agents a propose-then-review path for drafts (compose YAML, shell scripts, configs, markdown). The agent submits draft content + kind; the Tool runs the per-kind validator (today: composevalidator deny-list for compose-yaml) and returns a structured envelope the agent's UI can surface for operator approval. Nothing is persisted, nothing is executed — `install_app` remains the only mutation path. Pre-fix the agent had no way to propose a draft for review: chat prints skip the deny-list validator, install_app runs immediately. (P1.6 from 2026-05-31 MCP-only chat-mode test.)
- New `list_capabilities` MCP Tool reports which operator-opt-in tiers (destructive write tools, sensitive sysadmin resources) are active on this server. Agents call it before attempting a gated capability — saves a trial-and-error round-trip when an operator hasn't opted in. Includes a Summary string so even an agent that doesn't parse the structured fields surfaces the right state to the operator. (P2.8 from 2026-05-31 MCP-only chat-mode test.)
- CI gate `scripts/check-mcp-docs-canonical.sh` enforces ADR-0050's "MCP-served docs are canonical" doctrine on the compose-extension key family (x-powerlab / x-web / x-casaos). Every extension key the parser accepts MUST be mentioned in `docs/concepts/compose-conventions.md` — drift fails CI with a clear pointer to both files so a contributor adding a 4th alias can't ship without the doc update. Initial scope is the extension keys only; follow-up PRs extend the same shape to OpenAPI paths, x-powerlab subfield names, and catalog metadata.
### Changed
- `search_docs` MCP tool now indexes three documentation surfaces instead of just one — agents searching for "install_app" (OpenAPI), "vaultwarden" (catalog) or "compose-conventions" (concepts) get results from whichever surface the term lives in, with a `source` field on each hit so they know which canonical URI to chain to. Pre-fix the tool was concepts-only and returned `{matches: []}` for any term that lived in OpenAPI specs or the bundled catalog, even when the answer was clearly there. (P0.3 from 2026-05-31 MCP-only chat-mode test.)
- Cleanup pass on the MCP UX batch: `generate_artifact` now reports a three-state validation `status` ("ok" | "violations" | "skipped") instead of a confusing `ok: true|false` that conflated "validator rejected" with "no validator wired"; `get_system_health` uses stdlib `strconv` instead of hand-rolled integer/float formatters; `list_capabilities` ships with a drift-safety test that reconciles its hardcoded tier lists against the live server registrations, so any future PR that adds a destructive tool or sensitive resource fails CI until the operator-facing list is updated.
- `powerlab-mcp-smoke` now exercises the 6 new MCP Tools added in the 2026-06-01 UX batch (browse_catalog, get_compose_conventions, start_compose_authoring, get_system_health, generate_artifact, list_capabilities) with typed shape probes. Pre-fix the smoke only covered the original journal_search + check_disk_free, so a future refactor breaking any new Tool's output contract would have escaped to operators. Quickstart docs updated to list all 11 always-on tools (was 4) plus the 2 gated ones.
- `browse_catalog` and `search_docs` now return a `note` field on the zero-match case explaining how the matcher works (case-insensitive substring) and suggesting recovery (try a shorter fragment, omit filter, browse the index resource). Pre-fix, both tools returned a bare empty array with no breadcrumb, so an agent that got zero hits couldn't tell "filter too narrow" from "catalog/corpus empty" and had no path to retry productively. Caught by Batch B prompt #5 of the extended end-to-end test.
- composevalidator now allows `/dev/dri` device passthrough (the standard Intel/AMD/Pi VAAPI + V4L2-M2M HW transcode surface) for agent-installed apps. This unblocks catalog apps like jellyfin, emby, and frigate that ship with `devices: [/dev/dri:/dev/dri]` — previously agent-install rejected them while the UI install path accepted them, an inconsistency the Batch B end-to-end test flagged. All other `/dev/*` paths (nvidia raw, kmem, sda, ttyUSB*, …) stay rejected; widening the allowlist is a per-device decision with its own ADR.
### Fixed
- In-app upgrade no longer fails with "context deadline exceeded
(Client.Timeout or context cancellation while reading body)" when
downloading the tarball on slow / typical homelab connections.
Root cause: the updater's HTTP client set Client.Timeout=30s,
which counts the BODY read against the wall clock — an 80 MB
release tarball at ~2.7 MB/s (typical ISP) needs 25-40 s and
raced the budget mid-stream. Fix moves all bounds onto the
Transport (DialContext 10s, TLS 10s, ResponseHeader 30s,
IdleConn 90s) and lets the body read run as long as the caller's
context allows — operator-navigating-away still cancels cleanly,
but a healthy stream completes regardless of total size. Locked
by a regression test (positive: slow-stream succeeds; negative:
old-style Client.Timeout=1s fails with the exact bug error
message) so this never reappears silently.

- docker://volumes now populates `size` and `in_use_by` instead of always reporting `-1` and `[]`. The Docker SDK's VolumeList never populates UsageData regardless of filters, so the v0.7.6 implementation left every volume row stranded — the agent could answer "how much disk are volumes using overall?" via docker://system but could not answer "which volume is fat?" The fix calls /system/df (DiskUsage with the VolumeObject filter) for size enrichment and walks ContainerList's Mounts to compute in_use_by[]{id,name} — the join is scoped to Type=volume mounts so bind sources with a colliding name don't pollute the result. Both enrichments are best-effort: if /system/df or ContainerList fails the listing still returns 200 with degraded fields (size=-1, in_use_by=[]) so the agent never loses the volume names.
- Operator helper CLIs `powerlab-mcp-validate` and `powerlab-mcp-smoke`
now ship with the install. Before this fix the README "30-second smoke
test" and the MCP operator quickstart referenced both binaries but
`install.sh` only laid down `/usr/bin/powerlab-mcp`; every documented
command failed with "No such file or directory" on a fresh install.
`scripts/package-linux.sh` now cross-compiles both helpers
(cmd/validate + cmd/smoke) with the same ldflags + CGO_ENABLED=0
envelope as the main daemon and stages them into `$STAGE/bin/`, where
the existing wildcard `install -m 0755 "$HERE/bin/powerlab-"*`
installs them to `/usr/bin/powerlab-mcp-validate` and
`/usr/bin/powerlab-mcp-smoke` — operator PATH, alongside the main
binary. README + operator quickstart paths updated to `/usr/bin/`
(the previous `/usr/share/powerlab/bin/` was aspirational; that
directory never existed). Regression test
`scripts/check-package-linux-powerlab-mcp_test.sh` extended to lock
the cross-compile, the shared LDFLAGS_VERSION_STAMP, and the
wildcard install line so this contract can't quietly regress.

- **MCP — `journal://system/failures` no longer returns `exit status 1`.** The sensitive-tier resource shipped with `-p warning..error` in the journalctl argv; journalctl reads `-p LOW..HIGH` using the numeric syslog priorities (emerg=0..debug=7), so `warning..error` parsed as 4..3 — reversed — and on systemd ≥ 245 errored out with "Failed to parse log level range warning..error". Plus `error` is not even a valid priority name (the canonical short form is `err`). Corrected to `err..warning` (3..4, low-to-high), which is what ADR-0049 specified all along ("filtered to PRIORITY=warning..error"). Regression test `TestBuildSystemArgs_FailuresPriorityRangeIsValid` parses the emitted `-p` value and asserts `low <= high` numerically, so any future reversal (info..crit, warning..err, etc.) fails LOUD instead of waiting for a user-visible exit-1; a Linux-only integration test execs real journalctl with the produced args to catch the exact failure mode #639 reported.
- MCP audit records now carry real agent identity — JWT subject populates user_id/username even on loopback (Claude Desktop on the same box), and tools/call + resources/read enrich the record with kind + tool_name / uri so operators can answer 'which agent called which tool' (#644).
- jwt.HTTPJWT now recognises IPv6 loopback (`[::1]:PORT`) connections arriving via limactl / Docker Desktop forwarding. The bracketed IPv6 form was previously stripped to `[::1]` and compared against `"::1"` — so every Mac→VM request failed the loopback bypass and was rejected with `missing/invalid token` instead of being treated as a trusted local agent.
- `journal_search` MCP tool now returns clean, agent-readable error messages instead of leaking raw subprocess noise. The pre-fix shape was `journal read: run journalctl: exit status 1` (opaque on Mac dev boxes, non-systemd containers, broken installs); post-fix the agent sees `journald unavailable on this host (systemd-journald not running, or journalctl not in PATH — common on macOS dev boxes and non-systemd containers)`. Timeouts and ACL-denied paths also produce sanitized messages with hints. (P0.2 from 2026-05-31 MCP-only chat-mode test.)
- Audit middleware (HTTP-stdlib variant) and the gateway's rewriteRequestSourceIP now recognise IPv6 loopback (`[::1]:PORT`) connections that arrive via limactl / Docker Desktop forwarding. Same bug class as the jwt fix shipped in PR #653 (2026-05-31): the LastIndex(:) port strip left brackets in the result, so the `remote == "::1"` check missed every Mac→VM request. Audit was logging the raw `[::1]` instead of the LoopbackSentinel; the gateway was failing to recognise the loopback IP and skipping its X-Forwarded-For rewrite. Echo's `c.RealIP()`-based middleware was already correct (Echo handles brackets internally) and is unchanged.
- `get_system_health` now reports OS-update state correctly. The original P1.5 implementation called `coreproxy.GetFrom(ctx, "/v1/sys/updates", ...)` but that endpoint does not exist on core — `system://updates` reads `apt list --upgradable` output directly via the apt runner in resources_system_updates.go. The proxy call therefore failed with 404 on every Linux host, reporting `updates=unknown` regardless of pending package count. Smoke (added in PR #662) caught it on Lima. Fix: wire the same `collectUpdates` helper the resource already uses, with a testable apt-runner seam so the tests don't depend on the host having apt.
- `get_system_health` now parses the canonical PowerLab response envelope (`{success, message, data}`) and reads disk usage from the `mounts[]` array on VM hosts where `physical[]` is empty (Lima, Docker Desktop). Pre-fix, the Tool decoded the envelope as a flat object and saw empty arrays everywhere, reporting `worst mount: 0.0%` on Lima at 88.7% used + "all services healthy" regardless of state. Services also gained `.service` suffix tolerance — real core returns `powerlab-mcp` without the suffix; the evaluator now matches both forms. Caught by end-to-end testing in Claude Code post-deploy (the agent rendered "worst mount at 0.0%" when reality was 88.7%).
- `get_system_health` no longer grades pseudo-filesystems as disk pressure. Read-only mounts that are always full by design — Lima's iso9660 cloud-init seed, Snap squashfs loops, the kernel's tmpfs / proc / sysfs — are excluded from the worst-mount computation. Pre-fix a fresh Lima box reported `disk: critical (worst mount /mnt/lima-cidata at 100%)` while the real ext4 root at 88.7% was below the warn threshold. The agent talking to Claude Code spotted the artefact and flagged it on the first end-to-end probe of the just-fixed envelope parsing.
- `journal://system/auth` and `journal://system/failures` (ADR-0049 sensitive tier) now return real data. The hardcoded authSelectors lacked the `+` separator between the `_SYSTEMD_UNIT=` matchers and the `_COMM=` matchers, so journalctl ANDed across the two field groups instead of ORing them. The resulting filter `(_SYSTEMD_UNIT=ssh.service OR sshd.service) AND (_COMM=sudo OR su)` is impossible — sudo records never carry `_SYSTEMD_UNIT=ssh`. Both resources returned `[]` on every host since shipment. Caught by an agent in Claude Code that read the auth journal, found zero entries, traced the selectors and asked "are you sure?".
