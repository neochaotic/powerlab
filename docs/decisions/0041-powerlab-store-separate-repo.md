# 0041 — Split community catalog into separate `powerlab-store` repo

- **Status:** proposed
- **Date:** 2026-05-19
- **Supersedes (in part):** ADR-0039 ("PowerLab ships a native, vendored, curated catalog") — the **what** stays; the **where it lives** moves out of the core repo into a dedicated catalog repo.
- **Trigger:** Sprint 23 surfaced two pains the monolith made worse:
  1. **Hotspot file conflicts** — `community-catalog/Apps/` is touched by every catalog edit AND every code release; 20+ PRs in one day produced repeated rebases against this dir.
  2. **Catalog cadence ≠ code cadence** — operators want fresh app entries weekly; cutting a PowerLab core release just to ship a new app yaml is over-coupling.
- **Companion**: The reconstruction work (139-app artefact analysed 2026-05-19) sits on top of this split. Importing 139 apps into the monolith repeats the hotspot pain at scale; the split is the prerequisite.

## Context

The catalog has been part of the PowerLab core repo since the CasaOS rebrand. ADR-0039 (Sprint 22) made it native + curated; Sprint 23 PR #451 + #452 + #455 closed the install-path bugs. With 4 curated apps the friction is tolerable. With 139 apps, every PR that touches a single compose YAML is in conflict territory with every release cut.

External patterns: Kubernetes split `kubernetes/kubernetes` (core) from chart repositories + Artifact Hub. Helm splits the engine from the chart registry. Docker splits the daemon from Docker Hub. PowerLab is small but the pattern scales down cleanly.

The companion artifact (analysed in this same session, 2026-05-19) provides 139 vetted apps + scripts/validate.py + scripts/pin-digests.sh. Importing it AS-IS into the monolith means:

- 139 new directories under `community-catalog/Apps/` in a single PR (giant diff)
- Every future catalog edit collides with code release work
- Icons hot-linked to `getumbrel.github.io` instead of rehosted (memory `feedback_umbrel_icons_as_is` revisit trigger fires here)
- Catalog releases tied to PowerLab core release cadence

## Decision

PowerLab catalog moves to a separate repo: **`neochaotic/powerlab-store`** (already created, public, AGPL-3.0 base).

### Repo layout (`powerlab-store`)

```
powerlab-store/
├── README.md                    # what + why + how operators use it
├── LICENSE                      # AGPL-3.0 (overall, see "Licensing")
├── LICENSE-CONTENT              # CC-BY-SA-4.0 for Apps/ + icons/ (see "Licensing")
├── CONTRIBUTING.md              # app submission manual
├── SECURITY.md                  # disclosure process
├── CODE_OF_CONDUCT.md
├── docs/
│   ├── schema.md                # x-powerlab spec (canonical)
│   ├── curation-criteria.md     # what passes / what's rejected
│   ├── icon-guidelines.md       # size, format, attribution
│   └── adr/                     # ADRs SPECIFIC to the store repo
├── Apps/
│   └── <app-id>/
│       ├── docker-compose.yml   # with x-powerlab block, digest-pinned
│       ├── icon.svg             # or icon.png — rehosted
│       └── description.md       # long-form description
├── scripts/
│   ├── validate.py              # schema + safety gates (from artifact)
│   ├── check-safety.sh          # mirror of core's safety lint
│   ├── pin-digests.sh           # digest resolver (BSD/GNU sed safe)
│   └── build-index.sh           # emits index.json + tarball release asset
├── .github/
│   ├── workflows/
│   │   ├── validate.yml         # all gates on every PR
│   │   ├── release.yml          # tag → tarball + index.json release asset
│   │   └── icon-lint.yml        # SVG/PNG sanity + size cap
│   ├── ISSUE_TEMPLATE/
│   │   ├── new-app.md
│   │   └── app-bug.md
│   └── PULL_REQUEST_TEMPLATE.md
└── index.json                   # autogenerated catalog manifest
```

### Licensing

**Hybrid model** (rejected: AGPL-3.0 monolithic; reason: AGPL was designed for software with a network-use trigger; applying it as the sole license to YAML configuration + icons + descriptive prose is non-orthodox and would discourage contributions of art/text).

| Surface | License | Rationale |
|---|---|---|
| `scripts/` (validate.py, pin-digests.sh, build-index.sh) | **AGPL-3.0** | Aligns with PowerLab core; these are software. |
| `Apps/<id>/docker-compose.yml` | **AGPL-3.0** (as derivative configuration coupled to PowerLab core) | Defensible. Catalog is part of PowerLab's UX. |
| `Apps/<id>/description.md` | **CC-BY-SA-4.0** | Prose — natural fit. Attribution + share-alike preserves contributor credit. |
| `Apps/<id>/icon.svg` `icon.png` | **CC-BY-SA-4.0** OR original creator's license preserved | Art has its own attribution chain. Some upstream icons retain their original license — recorded in `x-powerlab.icon.license`. Newly-generated PowerLab-original SVG icons (geometric, monoline style — see [`docs/icon-guidelines.md`](https://github.com/neochaotic/powerlab-store/blob/main/docs/icon-guidelines.md)) are CC-BY-SA-4.0 by default. |
| `docs/` | **CC-BY-SA-4.0** | Documentation. |
| `index.json` | **AGPL-3.0** (generated artifact tracking software-side data) | Derived from `Apps/`. |

A top-level `LICENSE` carries AGPL-3.0 text; `LICENSE-CONTENT` carries CC-BY-SA-4.0 text. README's "License" section explains the split + per-surface mapping.

Reference pattern: Kubernetes ecosystem (code Apache-2.0, docs CC-BY-4.0, assets per-creator) — same approach, adapted to PowerLab's chosen AGPL stack.

### CI gates (mandatory on every PR)

1. **`validate.py`** — schema + ban hooks/exports.sh/env_file/getumbrel image/unrewritten upstream volume paths/cap_add SYS_ADMIN/ALL/network_mode host (already exists in the artifact, ported under AGPL).
2. **`check-safety.sh`** — mirror of `scripts/check-catalog-app-safety.sh` from PowerLab core; adds `ipc:host`, `pid:host`, system-path bind-mount bans that `validate.py` misses today.
3. **YAML lint** — `yamllint` with standard config; catches structural issues before they confuse validators.
4. **Digest pin gate** — **strict from day 1**: every `image:` MUST contain `@sha256:` or the PR fails. Unlike PowerLab core's govuln warn-only ratchet (ADR-0040), there's no excuse for unpinned images in a catalog repo — pinning IS the security model.
5. **Icon presence + size cap** — every `Apps/<id>/` MUST have `icon.svg` or `icon.png` ≤ 100KB.
6. **Schema completeness** — `x-powerlab.{store_app_id, title, tagline, category, port_map, source, icon, security}` required.
7. **Changie fragment** — every PR adds `.changes/unreleased/*.yaml` (same convention as core).
8. **Compose validity** — `docker compose -f docker-compose.yml config --quiet` parses without error.

These gates are **WARN-FREE strict from day 1** because the store starts with a clean baseline (4 apps migrated from core repo, vetted) — no legacy debt to ratchet through.

### Contract with PowerLab core

`powerlab-store` releases produce a single tarball asset per release tag:

```
powerlab-store-vN.M.P.tar.gz   # contains Apps/, icons/, index.json
```

PowerLab core consumes the catalog in **one of two modes**:

| Mode | When | How |
|---|---|---|
| **Bundled (default)** | New installs + air-gapped operators | `scripts/package-linux.sh` fetches latest `powerlab-store` release tarball at PowerLab build time and bundles it into the core tarball. Operators have a working catalog offline immediately on first install. |
| **External (operator override)** | Operators wanting bleeding-edge catalog OR a custom-fork store | Settings → Catalog → "Use external store URL". PowerLab periodically polls the store's `index.json` for newer releases and surfaces an in-app "Catalog update available" notification — separate from the PowerLab core update banner. |

Compat protocol (introduced in this ADR, implemented in v0.7.2+):

- Store releases declare `min_powerlab_version` in `index.json`
- PowerLab core declares `compatible_store_version_range` in its `manifest.json`
- Mismatch surfaces a clear "store too new / too old for this PowerLab" warning, never an opaque install error

### Migration phases (multi-release)

| Phase | Deliverable | Repo |
|---|---|---|
| 0 — bootstrap | This ADR + powerlab-store skeleton (README, LICENSE, CONTRIBUTING, validate.py + safety scripts, .github workflows, 1 reference app migrated) | both |
| 1 — reference content | Migrate the 4 current curated apps (adventurelog, baikal, enclosed, gitingest) from core repo to powerlab-store. Power on validation CI. Cut `powerlab-store-v0.1.0`. | powerlab-store |
| 2 — core consumer | PowerLab core `scripts/package-linux.sh` fetches `powerlab-store` release tarball at build time (bundled mode). UI Settings → Catalog reflects new origin label. | powerlab core (release-impact PR) |
| 3 — remove from core | Drop `community-catalog/` from PowerLab core repo. Update install scripts. Sprint 23 PR #451 wipe-then-copy now applies to the bundled-from-store tarball. | powerlab core |
| 4 — import 138 vetted apps | Add the 138 apps from the 2026-05-19 artifact (less openwebui — `ipc:host` removed or app dropped), with digest-pinning run on a Linux box (powerlab-store CI rejects unpinned images). | powerlab-store |
| 5 — descriptions + categorization pass | Rewrite all descriptions in consistent voice; collapse 13 ad-hoc categories down to 8 well-scoped categories. | powerlab-store |
| 6 — icons rehost | Migrate hot-linked Umbrel icon URLs to rehosted SVGs in `Apps/<id>/icon.svg`. Reference via jsdelivr CDN (`https://cdn.jsdelivr.net/gh/neochaotic/powerlab-store@vN/Apps/<id>/icon.svg`) for browser caching + zero rate limits. | powerlab-store |
| 7 — external mode | Implement Settings → Catalog "Use external store" UI + backend polling. Operators with internal forks can point to their own URL. | powerlab core |

Each phase ships independently. Phase order matters (e.g. Phase 4 needs Phase 1's CI gates landed first), but tempo doesn't (months between phases is fine).

## Consequences

### Positive

- **Hotspot eliminated** — catalog PRs don't touch core; core PRs don't touch catalog. The repeated-rebase pain of Sprint 23 doesn't recur as we import 138 apps.
- **Independent cadence** — catalog can ship weekly; core ships when core's ready.
- **Easier contribution** — outside contributors see a small focused repo, not a 50k-LOC monolith.
- **License clarity** — hybrid AGPL/CC-BY-SA is honest about what each surface is.
- **Memory `feedback_umbrel_icons_as_is` resolved** — rehost achieved at the right architectural moment (commercialization-grade catalog), not premature.
- **Memory `feedback_security_is_priority` reinforced** — store CI is strict-day-1 across ALL safety gates (no warn-only ratchets); the artefact's validate.py + our existing safety lint compose into a tighter floor.
- **Setpoint for ADR-0034 MCP**: the same repo-split pattern likely applies to the future observability service. ADR-0041 is precedent.

### Negative / accepted trade-offs

- **Two-repo coordination** — operators wanting to file a "catalog has wrong app" issue need to know which repo. Mitigation: cross-link in both READMEs + label scheme in core's issue tracker.
- **Bootstrap effort** — ~1 day setup for repo skeleton + CI + first reference app. One-time cost.
- **Store releases need their own cut discipline** — `scripts/cut-release.sh` pattern (v0.7.2 epic #483) needs to exist in `powerlab-store` too. Accept; same script can be vendored.
- **Versioning drift risk** — solved by `min_powerlab_version` / `compatible_store_version_range` compat protocol declared above.
- **No "single repo to rule them all"** for operators reading source — mitigation: core README links to store README prominently.

### Risks

- **Bundled vs external confusion** — operators may not realise their catalog source. Mitigation: Settings → Catalog UX clearly shows origin label + last sync timestamp (refines PR #458 work).
- **Store goes stale** if maintainer attention wanes — mitigation: bundled mode means PowerLab is never broken by a stale store; operators always have the latest-at-PowerLab-build catalog.
- **Icon CDN dependency** (jsdelivr) — falling back to GitHub raw URLs is a one-line config change.

## Rejected alternatives

- **Keep monolith** — repeats Sprint 23 hotspot pain; doesn't scale to 139+ apps; memory `feedback_umbrel_icons_as_is` revisit deferred indefinitely.
- **Monorepo with subtree split tooling** (git-subrepo / git-subtree) — adds tooling complexity, doesn't solve the hotspot problem (still one git history).
- **Catalog as a separate branch of core repo** — same git, same hotspot; rejected.
- **Catalog as a submodule** — submodule pain (operators forget `--recursive`, IDE confusion, partial checkouts). Has its place in C++/embedded land; over-engineered here.
- **Single AGPL-3.0 monolithic license** — discussed inline in Licensing section. Hybrid is more orthodox + contributor-friendly.

## Acceptance criteria

This ADR is **accepted** when:

1. `powerlab-store` repo created with the skeleton above (README, LICENSE+LICENSE-CONTENT, CONTRIBUTING, scripts/, .github/workflows/, 1 reference app migrated)
2. `powerlab-store-v0.1.0` release cut with 4 reference apps (Phase 1 deliverable)
3. PowerLab core PR lands switching install bundle source from `community-catalog/` to fetched `powerlab-store` tarball (Phase 2 deliverable)
4. Operator can install PowerLab + see Settings → Catalog correctly labeling the bundled source (Phase 2 verification)
5. Migration phases 3-7 tracked as separate issues/releases in their respective repos

If Phase 1-2 reveal the split is too operationally heavy, this ADR is amended (e.g. revert to monolith + add path-isolation tooling), not the implementation forced through.

### Cross-link maintenance

- **Permalinks over branches** in code references (`/blob/<sha>/` for ADRs that supersede; `/blob/main/` for evergreen ops docs)
- **Both READMEs link to each other on line 1-3** so a search engine surfacing one finds the other
- **CI gate (future)**: a periodic check that referenced URLs in markdown still resolve. Tracked as a v0.7.x candidate (low priority).
- **Memory `feedback_critique_before_executing`** applies — when proposing a change to either repo, the agent checks if the corresponding doc in the OTHER repo needs an update.
